# Spec: 3.12 — CatalogReader / CatalogWriter

## What to build (not how)

A read/write API for the three system catalog tables (`nexus_tables`,
`nexus_columns`, `nexus_indexes`) backed by heap pages allocated in 3.11.

This includes:

1. **Multi-page heap chain** — when a catalog heap page fills, a new page
   is allocated and linked, so the catalog grows without a hard limit.
2. **Auto-increment sequences** — unique, monotonically increasing `TableId`
   and `IndexId` generated by the engine, stored durably in the meta page.
3. **CatalogWriter** — DDL write operations (create/delete table, column, index)
   integrated with the WAL and MVCC RowHeaders.
4. **CatalogReader** — read operations with snapshot-based MVCC visibility
   (linear scan, no B+ Tree backing yet — see DEFERRED).

---

## System table IDs (WAL convention)

To distinguish catalog entries in the WAL without user table_id collisions:

```
SYSTEM_TABLE_TABLES  = u32::MAX - 2   // nexus_tables
SYSTEM_TABLE_COLUMNS = u32::MAX - 1   // nexus_columns
SYSTEM_TABLE_INDEXES = u32::MAX       // nexus_indexes
```

User `TableId`s start at 1 and grow upward. The `u32::MAX - N` range is
unreachable from the sequence, so there is no ambiguity.

## Heap chain — multi-page linked list

The `_reserved: [u8; 28]` field in `PageHeader` has bytes [0..8] used to
store `next_page_id: u64 LE` for heap pages in a chain.

- Value `0` = end of chain (no next page).
- Value `> 0` = page_id of the next heap page in the same catalog table.

The root page_id for each system table is read from the meta page via
`CatalogBootstrap::page_ids()`. Traversal always starts from the root.

### `HeapChain` (lives in `nexusdb-storage/src/heap_chain.rs`)

```rust
/// Multi-page heap: insert to last page, grow chain on full, scan all pages.
pub struct HeapChain;

impl HeapChain {
    /// Read `next_page_id` from a page's `_reserved[0..8]`.
    pub fn next_page(page: &Page) -> u64

    /// Write `next_page_id` into a page's `_reserved[0..8]`.
    /// Caller must call `page.update_checksum()` and `write_page` afterwards.
    pub fn set_next_page(page: &mut Page, next: u64)

    /// Insert `data` with `txn_id` into the chain rooted at `root_page_id`.
    ///
    /// Walks to the last page (next_page == 0). If that page is full,
    /// allocates a new `PageType::Data` page, chains it, and inserts there.
    ///
    /// Returns `(page_id, slot_id)` of the new tuple.
    pub fn insert(
        storage: &mut dyn StorageEngine,
        root_page_id: u64,
        data: &[u8],
        txn_id: u64,
    ) -> Result<(u64, u16), DbError>

    /// Delete a specific tuple by (page_id, slot_id) with MVCC stamp.
    pub fn delete(
        storage: &mut dyn StorageEngine,
        page_id: u64,
        slot_id: u16,
        txn_id: u64,
    ) -> Result<(), DbError>

    /// Returns all visible tuples in chain order (snapshot MVCC filter).
    /// Items: (page_id, slot_id, data_bytes).
    pub fn scan_visible(
        storage: &dyn StorageEngine,
        root_page_id: u64,
        snap: TransactionSnapshot,
    ) -> Result<Vec<(u64, u16, Vec<u8>)>, DbError>
}
```

`insert` invariant: the chain pointer `next_page` in `_reserved` must be
updated **before** `write_page` of the new page and the previous page.
Both writes must happen before the function returns — no partial chain state.

## Sequences — auto-increment IDs

The meta page body is extended with two `u32` sequence counters:

```
body[64..68]: next_table_id: u32 LE    (0 = uninitialized, 1 = first valid)
body[68..72]: next_index_id: u32 LE    (0 = uninitialized, 1 = first valid)
```

New constants in `nexusdb-storage/src/meta.rs`:

```
NEXT_TABLE_ID_BODY_OFFSET  = 64
NEXT_INDEX_ID_BODY_OFFSET  = 68
```

`CatalogBootstrap::init()` must write `next_table_id = 1` and
`next_index_id = 1` on fresh DB creation (idempotent).

API (free functions in `nexusdb-storage::meta`):

```rust
/// Reads next_table_id and atomically increments it in the meta page.
/// Returns the ID that was allocated (before increment).
pub fn alloc_table_id(storage: &mut dyn StorageEngine) -> Result<TableId, DbError>

/// Reads next_index_id and atomically increments it in the meta page.
pub fn alloc_index_id(storage: &mut dyn StorageEngine) -> Result<u32, DbError>
```

Both functions use the read-modify-write pattern already established by
`write_checkpoint_lsn` and `write_catalog_header`.

## CatalogWriter

Lives in `nexusdb-catalog/src/writer.rs`.

```rust
pub struct CatalogWriter<'a> {
    storage: &'a mut dyn StorageEngine,
    txn: &'a mut TxnManager,
    page_ids: CatalogPageIds,
}

impl<'a> CatalogWriter<'a> {
    pub fn new(
        storage: &'a mut dyn StorageEngine,
        txn: &'a mut TxnManager,
    ) -> Result<Self, DbError>

    /// Allocates a new TableId, inserts a row in nexus_tables, WAL-logs it.
    /// Returns the allocated TableId.
    pub fn create_table(&mut self, schema: &str, name: &str) -> Result<TableId, DbError>

    /// Inserts a row in nexus_columns. col_idx is set by the caller.
    pub fn create_column(&mut self, def: ColumnDef) -> Result<(), DbError>

    /// Allocates a new IndexId, inserts a row in nexus_indexes, WAL-logs it.
    /// Returns the allocated IndexId (u32).
    pub fn create_index(&mut self, def: IndexDef) -> Result<u32, DbError>

    /// Marks all rows in nexus_tables, nexus_columns, and nexus_indexes
    /// for table_id as deleted. WAL-logs each deletion.
    pub fn delete_table(&mut self, table_id: TableId) -> Result<(), DbError>

    /// Marks the matching row in nexus_indexes as deleted. WAL-logs it.
    pub fn delete_index(&mut self, index_id: u32) -> Result<(), DbError>
}
```

### WAL integration protocol

For each create/delete operation:
1. Perform the heap mutation (insert_tuple / delete_tuple).
2. Call `txn.record_insert(table_id, key, data, page_id, slot_id)` or
   `txn.record_delete(table_id, key, data, page_id, slot_id)`.
3. The caller is responsible for `commit()` or `rollback()`.

System table_ids for WAL entries use the `SYSTEM_TABLE_*` constants.
The key bytes encode the row's logical primary key (table_id as u32 LE, etc.).

## CatalogReader

Lives in `nexusdb-catalog/src/reader.rs`.

```rust
pub struct CatalogReader<'a> {
    storage: &'a dyn StorageEngine,
    page_ids: CatalogPageIds,
    snapshot: TransactionSnapshot,
}

impl<'a> CatalogReader<'a> {
    pub fn new(
        storage: &'a dyn StorageEngine,
        snapshot: TransactionSnapshot,
    ) -> Result<Self, DbError>

    /// Returns the first visible table matching (schema_name, table_name).
    pub fn get_table(&self, schema: &str, name: &str) -> Result<Option<TableDef>, DbError>

    /// Returns the first visible table with the given table_id.
    pub fn get_table_by_id(&self, table_id: TableId) -> Result<Option<TableDef>, DbError>

    /// Returns all visible tables in the given schema.
    pub fn list_tables(&self, schema: &str) -> Result<Vec<TableDef>, DbError>

    /// Returns all visible columns for table_id, ordered by col_idx.
    pub fn list_columns(&self, table_id: TableId) -> Result<Vec<ColumnDef>, DbError>

    /// Returns all visible indexes for table_id.
    pub fn list_indexes(&self, table_id: TableId) -> Result<Vec<IndexDef>, DbError>
}
```

All read methods use `HeapChain::scan_visible` with `self.snapshot` for
MVCC filtering. No B+ Tree backing — linear scan over heap pages.

## Use cases

1. **Fresh DB — first CREATE TABLE**: `init()` sets sequences to 1.
   `create_table("public", "users")` allocates table_id=1, inserts row, WALs it.
   Commit. `get_table("public", "users")` returns TableDef{id:1}.

2. **Multiple columns**: `create_column({table_id:1, col_idx:0, ..})`,
   `create_column({table_id:1, col_idx:1, ..})`. `list_columns(1)` returns
   both, ordered by col_idx.

3. **Heap chain grows beyond one page**: inserting enough rows that the root
   page fills (heap is `HeapPageFull`) causes `insert` to allocate a new page,
   chain it, and retry. Scan returns all rows across all pages.

4. **DROP TABLE**: `delete_table(table_id)` marks all matching rows deleted.
   A subsequent `get_table` with a newer snapshot returns None.

5. **Rollback DDL**: `create_table`, then `rollback()`. The table's row is
   marked dead (UndoInsert). A subsequent `get_table` with any committed
   snapshot returns None.

6. **Snapshot isolation**: snapshot taken before `create_table` does NOT see
   the new table. Snapshot taken after commit DOES see it.

7. **Multi-page scan after restart**: `MmapStorage` reopened. `CatalogReader`
   scans chain pages via `_reserved` pointers. All rows visible.

8. **Sequence persistence**: after `create_table` (table_id=3), crash recovery,
   reopen — `alloc_table_id` returns 4, not 1 (no collision).

9. **index_id auto-increment**: `create_index` allocates unique index_id;
   two consecutive calls return different IDs even if table_id is the same.

## Acceptance criteria

- [ ] `HeapChain::insert` handles `HeapPageFull` by chaining a new page
- [ ] `HeapChain::scan_visible` returns rows from all pages in chain order
- [ ] `HeapChain::delete` stamps `txn_id_deleted` on the target slot
- [ ] `next_page` in `_reserved[0..8]` survives a `write_page` round-trip
- [ ] `alloc_table_id` returns monotonically increasing IDs across calls
- [ ] `alloc_index_id` returns monotonically increasing IDs across calls
- [ ] Both sequences persist across `MmapStorage` close/reopen
- [ ] `CatalogBootstrap::init` initializes both sequences to 1
- [ ] `CatalogWriter::create_table` inserts a visible row and WAL-logs an Insert entry
- [ ] `CatalogWriter::create_column` inserts a visible row
- [ ] `CatalogWriter::create_index` inserts a visible row and WAL-logs an Insert entry
- [ ] `CatalogWriter::delete_table` marks all rows for that table_id as deleted
- [ ] `CatalogWriter::delete_index` marks the matching index row as deleted
- [ ] `CatalogReader::get_table` finds a table by (schema, name) or returns None
- [ ] `CatalogReader::get_table_by_id` finds by table_id or returns None
- [ ] `CatalogReader::list_columns` returns columns ordered by col_idx
- [ ] `CatalogReader::list_indexes` returns all indexes for a table_id
- [ ] Snapshot taken before commit does NOT see the new row
- [ ] Snapshot taken after commit DOES see the new row
- [ ] After rollback, the inserted row is not visible to any committed snapshot
- [ ] After `delete_table`, a newer snapshot does not see the deleted rows
- [ ] Multi-page catalog: after filling a page, new rows are inserted in a chained page and scanned correctly
- [ ] Sequences survive close/reopen (MmapStorage persistence)
- [ ] No `unwrap()` in `src/` (excluding tests and benches)

## ⚠️ DEFERRED

- **Index-backed catalog lookups** — B+ Tree on `(schema, name) → table_id`
  creates a bootstrap cycle (index registration requires the catalog writer).
  Deferred to Phase 4.x once the executor is in place. Linear scan is O(n)
  but correct for catalogs up to thousands of tables.
- **`col_idx` auto-assignment** — the caller provides `col_idx`; no enforcement
  of contiguous column ordering or gap detection. Deferred to Phase 4 executor.
- **Catalog cache** — no in-memory cache of TableDef/ColumnDef. Every read
  scans heap pages. Deferred to Phase 5 (session state caching).
- **Catalog change notifier** (3.13) — pub-sub for DDL changes. Builds on top
  of the writer in this subphase.

## Out of scope

- Schema validation (duplicate table name detection) — no unique constraint on
  the catalog yet; enforced by the executor in Phase 4.18
- `RENAME TABLE` / `ALTER TABLE` — Phase 4.22
- Catalog metrics / `information_schema` — Phase 4.20
- Concurrent DDL (Phase 7 MVCC)
